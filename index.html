Introduction

C++ is a compiled, object-oriented, general-purpose programming language. It is used by a wide range of organizations around the world. When developing programs in C++, it is important to follow a consistent coding style to ensure that the code is easy to debug, read, and maintain. The purpose of this guide is to provide an overview of the coding practices we follow when developing software in C++ and why we believe they are correct. It is important to note that this is not a comprehensive C++ tutorial, and we assume that the reader is already familiar with the language.

 

Version C++

As of 2025, the code should be written using the features of C++ 20. It is consistently supported by a larger number of compilers than C++ 23. For comparison, refer to the standard tables on the website cppreference.

Try to minimize the use of non-standard language extensions in your code. These include GCC attribute, built-in functions such as __builtin_prefetch or SIMD, #pragma directives, inline assembly, macros such as COUNTER or PRETTY_FUNCTION, compound expressions (for example, foo = ({ int x; Bar(&x); x })), variable-length arrays, functions such as alloca(), and the a?.

 

Although non-standard extensions may provide useful features that are not present in standard C++ and may be necessary to optimize compiler performance, they are not always supported by all compilers. Using such extensions reduces the portability of the code, as they may work differently in different compilers. Even if an extension is supported by all target compilers, its implementation may differ, leading to ambiguities in behavior. The use of non-standard extensions complicates the code, requiring the developer to know these extensions in order to understand it and transfer it to other architectures. Therefore, do not use non-standard extensions, except when you use them in wrappers that provide portability. In this case, such wrappers should be included in a dedicated header for the entire project.

 

Comments

Comments are extremely important for maintaining code readability. Here are some basic rules for writing them. However, remember that while comments are very useful, the best code is the code that explains itself. It is better to use clear names for types and variables than to explain unclear names in comments. When writing comments, focus on your audience: the people who will be reading your code later. Be generous with your comments — you might be reading them yourself in the future!

 

Punctuation, spelling, and grammar

Pay attention to the punctuation, spelling, and grammar in your comments. Well-written comments are easier to read than poorly written ones. Comments should be easy to read, like regular text, with proper use of capital letters and punctuation. Full sentences are usually easier to read than fragments of phrases. Short comments, such as notes at the end of a code line, can be less formal, but try to maintain a consistent style. While it may not be pleasant when a reviewer points out a punctuation error, it's important to keep the code clear and understandable. Proper use of punctuation, spelling, and grammar contributes to this.

 

Comment style

You can use both single-line comments // and multi-line comments /* */, but it is important to maintain consistency. Although both styles of comments are acceptable, // is more commonly used. Be consistent in choosing the comment style and applying it to your code.

 

Comments on files

When creating each header file, use the license template. If the source file (for example, an .hpp file) contains several custom abstractions (such as common functions or related classes), add a comment that describes these abstractions. The comment should be detailed enough so that future developers can understand what is contained in the file. However, more detailed documentation on individual abstractions should be found directly in their descriptions, and not in the comments to the file. For example, if you are writing a comment to the Frobber.hpp file, you do not need to add a similar comment to the files. Frobber.cpp or Frobber.test.hpp. At the same time, if you have a file, for example RegisteredObjects.cpp containing a collection of classes without an associated header file, add a comment to this file to explain its contents.

 

The official notification and the author's line

Each header file must contain a license template. Make sure you use the correct template for the license used in your project (e.g., Apache 2.0, BSD, LGPL, GPL). If you make significant changes to a file that includes "Author", consider removing this line. New files typically do not require copyright notices or "Author" lines.

 

Comments on issues

Use TODO comments for temporary solutions, short-term fixes, or for cases where the code is not perfect yet, but it works. TODO comments should contain the word TODO in capital letters, as well as the error ID, name, email address, or other communication method that will help you better understand the problem.

Examples:

// TODO: Error 12345678 - delete after the window expires

// compatibility 2047q4.

// TODO (Ivan): Use "\*" here to concatenate strings.

 

If your TODO includes a plan for the future, specify either a specific date ("fix by November 2025") or an event ("remove this code when all clients can handle XML responses").

 

Naming

The most important consistency rules are those that govern naming. The name style immediately tells us what kind of named object it is: type, variable, function, constant, macro, etc., without requiring us to search for the declaration of this object. The mechanism of selecting patterns in our brain largely depends on these naming rules. The naming rules are pretty arbitrary, but we believe that consistency in this area is more important than individual preferences, so whether you consider them reasonable or not, rules are rules.

 

File Names

All file names must be written in Camel Case. Follow the rules used in your project. If there is no consistent local pattern to follow, give preference to this case. Example of a valid file name: MyUsefulClass.cpp

 

C++ files must end in .cpp, and header files must end in .hpp. Files that rely on text inclusion in certain locations must end in .inc. In general, make the file names very specific. For example, use HttpServerLogs.hpp instead of Logs.hpp. A very common case is the presence of a pair of files named, for example, FooBar.hpp and FooBar.cpp , defining a class named FooBar.

 

Mixing C and C++ header files can lead to confusion:

Formatting the code.

The IDE often adjusts formatting for .h and .hpp files are different. If used .h for C++ headers, you can accidentally apply C styling.

Confusion in naming.

C and C++ headers with similar names (for example, MyClass.h and MyClass.hpp) help you quickly distinguish between C and C++ files, which is especially important when using C wrappers for libraries.

Errors when connecting C headers in C++.

When linking C headers, you need to wrap them in extern "C" to avoid call conflicts due to different name mangling. If you don't do this, you may encounter compilation errors.

 

Type Names

Type names begin with a capital letter and contain a capital letter for each new word without an underscore: MyExcitingClass, MyExcitingEnum. All type names—classes, structures, type aliases, enumerations, and type template parameters—follow the same naming convention. Type names must begin with a capital letter and contain a capital letter for each new word. There are no underscores.

Examples:

// Classes and Structs

class UrlTable{};

class UrlTableTester{};

struct UrlTableProperties{};

 

// Typedefs

typedef hash_map<UrlTableProperties*, std::string> PropertiesMap;

 

// using aliases

using PropertiesMap = hash_map<UrlTableProperties*, std::string>;

 

// enums

enum class UrlTableError{};

 

Function names

The names of public functions of a class or structure begin with a capital letter, each word in the name also begins with a capital letter, underscores are not used: MyFunction(). The names of private functions of a class or structure begin with the character "_", the rest of the name is formed similarly to public functions. If the function is outside the class, it should be referred to as public.

 

Names of function arguments and variables

The words in the function argument names must always be written in lowercase, separated by the "_", and end with "_a". Similarly, variables end with "_".

 

Formatting

Coding style and formatting do not have strict standards, but using a consistent style makes it easier to maintain control over the project. Team members may have different preferences for formatting rules, and some may require adaptation, but it is important that everyone follows a consistent style so that everyone can easily read and understand each other's code.

 

Line length

Lines of code should not exceed 80 characters, as this is the standard that most codes follow. Proponents of the rule claim that it helps when working with multiple windows where there is no room for expansion. Opponents believe that the 80-character limit is outdated and makes the code less readable on modern screens. Exceptions are allowed for long comments, string literals, include instructions, header protection, and using declarations that cannot be broken down without loss of convenience. In such cases, Clang-Format may be disabled.

 

Characters other than ASCII

Characters that are not included in the ASCII set should be used rarely and only in UTF-8 format. It is advisable not to embed the user's text directly into the source code, even if it is in English, so characters other than ASCII should be used minimally. However, in some cases it may be justified. For example, if your code works with data from files from foreign sources, it makes sense to encode non-ASCII strings if they are used as delimiters in this data. Similarly, test codes that do not need localization may include such strings. In such situations, it is better to use UTF-8, since this encoding is supported by most tools that work not only with ASCII. It is acceptable and even recommended to use hexadecimal encoding where it improves readability. For example, "\XeF\xBB\xBF" or "\uFEFF" denote a zero-width character without a Unicode break, which would be invisible if it were directly included in the source code as UTF-8. If possible, avoid using the u8 prefix, as its behavior has changed significantly in C++20 compared to C++17. You should not use the char16_t and char32_t types, as they are intended for non-UTF-8 texts. For the same reason, avoid wchar_t unless you are writing code to work with the Windows API, where this type is widely used.

 

Spaces and indents

Separating curly braces with spaces helps to improve the readability of the code. When the code is designed with the correct indents and spaces, it is easier for programmers to understand the structure and logic of the program. This is especially important when working in a team or when reading someone else's code.

Not recommended:

int x_ = 3, y_ = 7; double z_ = 4.25; x_++;

if (a_ == b_)

{

    Foo();

}

Recommended:

int x_ = 3;

int y_ = 7;

double z_ = 4.25;

x_++;

if (a_ == b_)

{

    Foo();

}

 

Function definitions and declarations

Returns the type of the function in the same line where its name is specified. Also place the parameters on the same line if they all fit. If the list of parameters is too long, move it to new lines in the same way as you do with arguments when calling a function.

The functions are designed as follows:

ReturnType ClassName::FunctionName(Type par_name1_a, Type par_name2_a)

{

    DoSomething();

}

 

If you have too much text to fit in one line:

ReturnType ClassName::ReallyLongFunctionName(Type par_name1_a,

                                             Type par_name2_a,

                                             Type par_name3_a)

{

    DoSomething();

}

 

Or if you can't even specify the first parameter:

ReturnType LongClassName::ReallyReallyReallyLongFunctionName(Type par_name1_a,

                                                             Type par_name2_a,

                                                             Type par_name3_a)

{

    DoSomething();

}

 

Some points to pay attention to:

• Choose appropriate parameter names.

• The parameter name can only be omitted if the parameter is not used in the function definition.

• If you can't fit the return type and the function name in one line, take a break between them.

• If you make a break after the return type of the declaration or function definition, do not indent.

• The open parenthesis is always in the same line as the function name. 

• There should never be a space between the function name and the open parenthesis.

• There should never be a space between parentheses and parameters.

• An open curly brace is always at the end of the last line of a function declaration, not at the beginning of the next line.

• The closing curly brace is either in the last line by itself, or in the same line as the opening curly brace.

• There must be a space between the closing parenthesis and the opening curly brace.

 • All parameters should be aligned as much as possible.

• The default indentation is 2 spaces.

• The parameters enclosed in the shell are indented by 4 spaces. Unused parameters that are obvious from the context can be omitted:

class Foo

{

public:

    Foo(const Foo&) = delete;

    Foo& operator=(const Foo&) = delete;

};

 

Formatting namespaces

The contents in the namespace are written without indentation.

The namespace does not add padding. For example:

Not recommended:

namespace

{

  // Mistake! There is an indentation where it is not necessary

    void Foo()

    {

        …

    }

 

} // namespace

 

Don't indent the namespace.:

Recommended:

namespace

{

    void Foo()

    { // Good. Without additional indentation

        ...

    }

} // namespace

 

Formatting classes

Place the sections in the following order: public, protected, and private. 

The indentation is four spaces. Separate the functions from the variables in sections.

For example:

Not recommended:

class MyClass

{

public:

    void Foo();

 

private:

    void Bar();

    int x_ = 0;

    int y_ = 0;

};

Recommended:

class MyClass

{

public:

    void Foo();

 

private:

    void Bar();

 

private:

    int x_ = 0;

    int y_ = 0;

};

 

Floating-point literals

Floating-point literals should always have a starting point with digits on both sides, even if they use an exponential number system. Readability is improved if all floating-point literals take this familiar form, as it helps ensure that they are not mistaken for integer literals, and that the E/e exponential notation is not mistaken for a hexadecimal digit. You can initialize a floating-point variable with an integer literal (provided that the type of the variable can accurately represent this integer), but note that a number in the exponential notation is never an integer literal.

Example:

Not recommended:

float f_ = 1.f;

long double ld_ = -.5L;

double d_ = 1248e6;  

Recommended:

float f_ = 1.0f;

float f2_ = 1.0;

float f3_ = 1;

long double ld_ = -0.5L;

double d_ = 1248.0e6;

 

Logical expressions

If you have a boolean expression that is longer than the standard string length, be consistent in breaking down the strings. In this example, the logical AND operator is always at the end of the lines.:

if (this_one_thing_ > this_other_thing_ &&

    a_third_thing_ == a_fourth_thing_   &&

    yet_another_ && last_one_)

{

    ...

}

Note that when wrapping the code in this example, both && logical AND operators are at the end of the line. Although moving all operators to the beginning of a line is also allowed. Feel free to intelligently insert additional parentheses, because when used correctly, they can be very useful for improving readability, but be careful when overusing them. Also note that you should always use punctuation operators such as && and ~, rather than verbal operators such as and and compl.

 

In the example above, curly brackets are separated by spaces, which makes the code visually divided into logical blocks. This approach allows you to quickly see where the block of code associated with the if condition begins and ends. This reduces the likelihood of errors and helps to maintain uniformity in the style of writing code, which is important for large projects and collective development. Put spaces between operators and operands:

int x_ = (a_ + b_) * c_ / d_ + Foo(); When the string becomes longer than 100 characters, divide it into two, making a newline after the operator, and continue writing:

 int result_ = ReallyLongFunctionOne() + ReallyLongFunctionTwo()

             + ReallyLongFunctionThree() + ReallyLongFunctionFour();

 

int result2_ = ReallyLongFunction(parameter_one_a,

                                  parameter_two_a,

                                  parameter_three_a,

                                  parameter_four_a,

                                  parameter_five_a,

                                  parameter_six_a);

 

Leave blank lines between functions and between groups of expressions.

void Foo()

{

    ...

}

 

void Bar()

{

    ...

}

 

Basic C++ Expressions

C++ is built on the basis of C, so you can solve problems using C++, and in the traditional way, using C. For example, to output text to the console, you can use the C++-style `cout` output operator, or the C-style global `printf` function:

Not recommended:

printf("Hello, world!\n");

Recommended:

std::cout << "Hello, world!" << std::endl;

 

If you are in doubt whether to use the `for` or `while` loop, remember: use `for` when the number of repetitions is known, and `while` when it is unknown:

Repeats 'size_' times:

for (int i_ = 0; i_ < size_; i_++)

{

    ...

}

Repeats as long as there is input:

std::string str_;

while (input_ >> str_)

{

    ...

}

When using control operators such as `if/else`, `for`, `while`, always wrap the body of these operators in curly braces `{}` and observe indentation, even if the body consists of a single line:

Not recommended:

if (size_ == 0)

    return;

else

{

    for (int i_ = 0; i_ < 10; i_++)

        std::cout << "ok" << std::endl;

}

Recommended:

if (size_ == 0)

{

    return;

}

else

{

    for (int i_ = 0; i_ < 10; i_++)

    {

        std::cout << "ok" << std::endl;

    }

}

Avoid using `break` and `continue`, except when absolutely necessary. Do not use the `exit` function, which terminates the program immediately. The program should terminate naturally via the 'return` operator in the `main' function. When using the 'if/else` operators, choose the appropriate constructions depending on the conditions.

 

Avoid multiple independent checks:

Not recommended:

if (grade_ >= 90)

{

    std::cout << "You got an A!";

}

if (grade_ >= 80 && grade_ < 90)

{

    std::cout << "You got a B!";

}

if (grade_ >= 70 && grade_ < 80)

{

    std::cout << "You got a C!";

}

Recommended:

if (grade_ >= 90)

{

    std::cout << "You got an A!";

}

else if (grade_ >= 80)

{

    std::cout << "You got a B!";

}

else if (grade_ >= 70)

{

    std::cout << "You got a C!";

}

 

If you have an `if/else`, return the result of the check directly.:

Not recommended:

if (score1_ == score2_)

{

    return true;

}

else

{

    return false;

}  

Рекомендуется:

return score1_ == score2_;

 

 

 

Не проверяйте значения логического типа, используя `==` или `!=` с `true` или `false`:

Не рекомендуется:

if (x_ == true)

{

    ...

}

else if (x_ != true)

{

    ...

}

Recommended:

if (x_)

{

    ...

}

else

{

    ...

}

 

Excess

If you use the same code twice or more, then find a way to remove the redundant code so that it doesn't repeat itself. For example, it can be placed in an auxiliary function. If the code being repeated is similar, but not quite, then try to make an auxiliary function that accepts parameters and represents the different part.:

Not recommended:

Foo();

x_ = 10;

y_++;

...

Foo();

x_ = 15;

y_++;

Recommended:

Helper(10);

Helper(15);

...

void Helper(int new_x_a)

{

    Foo();

    x_ = new_x_a;

    y_++;

}

 

Move the general code out of the if/else expression so that it doesn't repeat:

Not recommended:

if (x_ < y_)

{

    Foo();

    x_++;

    std::cout << "Привет!";

}

else

{

    Foo();

    y_++;

    std::cout << "Привет!";

}

Recommended:

Foo();

if (x_ < y_)

{

    x_++;

}

else

{

    y_++;

}

std::cout << "Привет!";

 

 

 

Designing classes

Encapsulation. Separate your objects by making all data fields in your class private:

class Student

{

private:

    int homework_score_;

    ...

}

 

.hpp vs .cpp. Always place class declarations and their parts in your own files, ClassName.hpp. Always collapse the class declaration files .hpp in the preprocessor block #if defined() / define / endif to avoid multiple declarations of the same class:

Point.hpp

#ifndef POINT_HPP

#define POINT_HPP

class Point

{

public:

    Point(int x_a, int y_a);

    int GetX() const;

    int GetY() const;

    void Translate(int dx_a, int dy_a);

 

private:

    int m_x_;

    int m_y_;

};

#endif

 

Point.cpp

#include "Point.hpp"

Point::Point(int x_a, int y_a)

{

    m_x_ = x_a;

    m_y_ = y_a;

}

 

void Point::Translate(int dx_a, int dy_a)

{

    m_x_ += dx_a;

    m_y_ += dy_a;

}

...

 

class vs struct. Always use class unless you're creating a very small and simple data type that only requires a few public variables and possibly a constructor to initialize them. Avoid unnecessary fields. Use fields to store meaningful information about your objects rather than temporary values that are only used once.

 

Functions and procedural design

A well-designed function should have the following qualities:

v Performs a specific task completely and without deviations.

v Does not take on unnecessary work.

v Does not interact with other functions unnecessarily.

v Efficiently uses memory to store data.

v Helps to structure the program and make it more clear.

v Helps to eliminate unnecessary code that might otherwise appear.

Use parameters to pass data to a function or return multiple values. Do not add parameters unless necessary. For example, in the following function, the variables `a`, `b`, and `c` are not parameters because they are not required:

void Quadratic(double a_a, double b_a, double c_a, double& root1_a,

               double& root2_a)

{

    double d_ = sqrt(b_a * b_a - 4 * a_a * c_a);

    root1_a = (-b_a + d_) / (2 * a_a);

    root2_a = (-b_a - d_) / (2 * a_a);

}

If you need to return a value, it is preferable to use `return`:

Not recommended:

void Max(int a_a, int b_a, int& result_a)

{

    result_a = (a_a > b_a) ? a_a : b_a;

}

Recommended:

int Max(int a_a, int b_a)

{

    return (a_a > b_a) ? a_a : b_a;

}

 

When passing objects to a function, try to use references, as this saves memory by avoiding copying objects. Give preference to reference variables instead of pointers, since references cannot be NULL:

Not recommended:

void Process(BankAccount* account_a)

{

    ...

}

Recommended:

void Process(BankAccount& account_a)

{

    ...

}

 

If the object will not be modified in the function, pass it as a `const` reference:

Not recommended:

void Display(BankAccount account_a)

{

    ...

}

Recommended:

void Display(const BankAccount& account_a)

{

    ...

}

 

Avoid long chains of function calls, when one function calls another, and that calls the next one, without returning values to `main'. Aim for the `main` to give a general idea of the program.

Введение

C++ — компилируемый объектно-ориентированный язык программирования общего назначения. Его используют огромное количество организаций по всему миру. При разработке программ на C++ важно придерживаться определённого стиля, чтобы в дальнейшем можно было без проблем отлаживать, читать и поддерживать код. Цель этого руководства — рассказать о правилах, которые мы придерживаемся при разработке программного обеспечения на C++ и считаем их правильными. Отметим, что это не учебное пособие по C++, и мы предполагаем, что читатель уже знаком с этим языком.

 

Версия C++

На момент 2025 года код должен быть написан с использованием возможностей C++ 20. Его стабильно поддерживает большее количество компиляторов, чем C++ 23. Для сравнения ознакомьтесь с таблицами стандартов на сайте cppreference.

Постарайтесь минимизировать использование нестандартных расширений для языка в вашем коде. К таковым относятся GCC attribute, встроенные функции типа __builtin_prefetch или SIMD, директивы #pragma, встроенные ассемблерные вставки, макросы вроде COUNTER или PRETTY_FUNCTION, составные выражения (например, foo = ({ int x; Bar(&x); x })), массивы переменной длины, функции вроде alloca(), и оператор a?.

 

Хотя нестандартные расширения могут предоставлять полезные функции, отсутствующие в стандартном C++, и могут быть необходимы для оптимизации производительности компилятора, они не всегда поддерживаются всеми компиляторами. Использование таких расширений снижает переносимость кода, поскольку они могут работать по-разному в различных компиляторах. Даже если расширение поддерживается всеми целевыми компиляторами, его реализация может отличаться, что приводит к неоднозначностям в поведении. Применение нестандартных расширений усложняет код, требуя от разработчика знания этих расширений для его понимания и переноса на другие архитектуры. Поэтому не используйте нестандартные расширения, за исключением случаев, когда вы используете их в обертках, которые обеспечивают переносимость. В таком случае такие обертки должны быть включены в специально отведенный заголовок для всего проекта.

 

Комментарии

Комментарии крайне важны для поддержания читаемости кода. Вот несколько основных правил для их написания. Однако помните, что, хотя комментарии очень полезны, наилучший код — это код, который сам объясняет себя. Лучше использовать понятные имена для типов и переменных, чем потом объяснять непонятные имена в комментариях. При написании комментариев ориентируйтесь на свою аудиторию: на тех, кто будет разбираться в вашем коде позже. Будьте щедры в комментариях — в будущем это можете быть вы сами!

 

Пунктуация, орфография и грамматика

Обращайте внимание на пунктуацию, орфографию и грамматику в комментариях. Хорошо написанные комментарии легче читать, чем плохо написанные. Комментарии должны быть легко читаемыми, как обычный текст, с правильным использованием заглавных букв и знаков препинания. Полные предложения обычно легче воспринимаются, чем обрывки фраз. Короткие комментарии, такие как примечания в конце строки кода, могут быть менее формальными, но старайтесь поддерживать единый стиль. Хотя это может быть не очень приятно, когда рецензент указывает на ошибку в пунктуации, важно, чтобы код оставался ясным и понятным. Правильное использование пунктуации, орфографии и грамматики способствует этому.

 

Стиль комментариев

Вы можете использовать как однострочные комментарии //, так и многострочные /* */, но важно соблюдать единообразие. Хотя оба стиля комментариев допустимы, // чаще используется. Будьте последовательны в выборе стиля комментариев и его применении в коде.

 

Комментарии к файлам

При создании каждого заголовочного файла используйте шаблон лицензии. Если исходный файл (например, .hpp файл) содержит несколько пользовательских абстракций (таких как общие функции или связанные классы), добавьте комментарий, который описывает эти абстракции. Комментарий должен быть достаточно подробным, чтобы будущие разработчики могли понять, что содержится в файле. Однако более детальная документация по отдельным абстракциям должна находиться непосредственно в их описаниях, а не в комментарии к файлу. Например, если вы пишете комментарий к файлу Frobber.hpp, вам не нужно добавлять аналогичный комментарий в файлы Frobber.cpp или Frobber.test.hpp. В то же время, если у вас есть файл, например RegisteredObjects.cpp, содержащий коллекцию классов без связанного заголовочного файла, добавьте комментарий к этому файлу, чтобы объяснить его содержание.

 

Официальное уведомление и строка автора

Каждый заголовочный файл должен содержать шаблон лицензии. Убедитесь, что вы используете правильный шаблон для лицензии, принятой в вашем проекте (например, Apache 2.0, BSD, LGPL, GPL). Если вы вносите значительные изменения в файл с указанием "Автор", подумайте о том, чтобы удалить эту строку. Новые файлы обычно не требуют уведомлений об авторских правах или строки "Автор".

 

Комментарии по задачам

Используйте комментарии TODO для временных решений, краткосрочных исправлений или для случаев, когда код ещё не идеален, но работает. TODO-комментарии должны содержать слово TODO заглавными буквами, а также идентификатор ошибки, имя, адрес электронной почты или другой способ связи, который поможет лучше понять проблему.

Примеры:

// TODO: ошибка 12345678 - удалить после окончания срока действия окна

// совместимости 2047q4.

// TODO (Иван): используйте "\ *" здесь для конкатенации строк.

 

Если ваш TODO включает в себя план на будущее, укажите либо конкретную дату ("исправить до ноября 2025 года"), либо событие ("удалите этот код, когда все клиенты смогут обрабатывать XML-ответы").

 

Именование

Наиболее важными правилами согласованности являются те, которые регулируют именование. Стиль имени сразу сообщает нам, что это за объект с именем: тип, переменная, функция, константа, макрос и т.д., не требуя от нас поиска объявления этого объекта. Механизм подбора шаблонов в нашем мозгу во многом зависит от этих правил именования. Правила именования довольно произвольны, но мы считаем, что последовательность в этой области важнее индивидуальных предпочтений, поэтому независимо от того, считаете ли вы их разумными или нет, правила есть правила.

 

Имена файлов

Все имена файлов должны быть написаны в регистре Camel Case. Следуйте правилам, используемым в вашем проекте. Если нет согласованного локального шаблона для подражания, отдайте предпочтение этому регистру. Пример допустимого имени файла: MyUsefulClass.cpp

 

Файлы C ++ должны заканчиваться на .cpp, а заголовочные файлы - на .hpp. Файлы, которые полагаются на текстовое включение в определенных местах, должны заканчиваться на .inc. В общем, делайте имена файлов очень конкретными. Например, используйте HttpServerLogs.hpp вместо Logs.hpp. Очень распространенным случаем является наличие пары файлов с именами, например, FooBar.hpp и FooBar.cpp, определяющих класс с именем FooBar.

 

Смешивание заголовочных файлов C и C++ может привести к путанице:

Форматирование кода.

В IDE часто настраивают форматирование под .h и .hpp файлы по-разному. Если использовать .h для C++-заголовков, можно случайно применить стилизацию C.

Путаница в имёнованиях.

Заголовки C и C++ с похожими именами (например, MyClass.h и MyClass.hpp) помогают быстро различать файлы для C и C++, что особенно важно при использовании обёрток для библиотек на C.

Ошибки при подключении C-заголовков в C++.

При подключении заголовков на C нужно оборачивать их в extern "C", чтобы избежать конфликтов вызовов из-за различного манглинга имён. Если не делать этого, могут возникнуть ошибки компиляции.

 

Имена типов

Имена типов начинаются с заглавной буквы и содержат заглавную букву для каждого нового слова без подчеркивания: MyExcitingClass, MyExcitingEnum. Имена всех типов — классы, структуры, псевдонимы типов, перечисления и параметры шаблона типа — имеют одинаковое соглашение об именовании. Имена типов должны начинаться с заглавной буквы и содержать заглавную букву для каждого нового слова. Никаких подчеркиваний.

Например:

// Classes and Structs

class UrlTable{};

class UrlTableTester{};

struct UrlTableProperties{};

 

// Typedefs

typedef hash_map<UrlTableProperties*, std::string> PropertiesMap;

 

// using aliases

using PropertiesMap = hash_map<UrlTableProperties*, std::string>;

 

// enums

enum class UrlTableError{};

 

Имена функций

Имена публичных функций класса или структуры начинаются с заглавной буквы, каждое слово в имени также начинается с заглавной буквы, подчёркивания не используются: MyFunction(). Имена приватных функций класса или структуры начинаются с символа "_", остальная часть имени формируется аналогично публичным функциям. Если функция находится вне класса, её следует именовать как публичную.

 

Имена аргументов функции и переменных

Слова в именах аргументов функции всегда должны писаться в нижнем регистре, отделяться символом "_" и иметь окончание "_a". Аналогично с переменными, но они оканчиваются на "_".

 

Форматирование

Стиль кодирования и форматирование не имеют строгих стандартов, однако использование единого стиля значительно облегчает контроль над проектом. Участники команды могут иметь разные предпочтения в отношении правил форматирования, и некоторые из них могут потребовать адаптации, но важно, чтобы все следовали определённому стилю, чтобы каждый мог легко читать и понимать код каждого пользователя.

 

Длина строки

Строки кода не должны превышать 80 символов, так как это стандарт, которому следует большинство кодов. Сторонники правила утверждают, что оно помогает при работе с несколькими окнами, где нет места для расширения. Противники считают, что ограничение в 80 символов устарело и делает код менее читаемым на современных экранах. Исключения допускаются для длинных комментариев, строковых литералов, инструкций include, защиты заголовков и using-объявлений, которые нельзя разбить без потери удобства. В таких случаях Clang-Format может быть отключён.

 

Символы, отличные от ASCII

Символы, которые не входят в набор ASCII, должны использоваться редко и только в формате UTF-8. Желательно не встраивать текст пользователя напрямую в исходный код, даже если это текст на английском, поэтому символы, отличные от ASCII, следует использовать минимально. Однако в некоторых случаях это может быть оправдано. Например, если ваш код работает с данными из файлов иностранных источников, имеет смысл закодировать строки, не входящие в ASCII, если они используются в этих данных как разделители. Аналогично, тестовые коды, которые не нуждаются в локализации, могут включать такие строки. В таких ситуациях лучше использовать UTF-8, поскольку эта кодировка поддерживается большинством инструментов, которые работают не только с ASCII. Допустимо и даже рекомендуется использовать шестнадцатеричную кодировку там, где это улучшает читаемость. Например, "\xEF\xBB\xBF" или "\uFEFF" обозначают символ нулевой ширины без разрыва в Юникоде, который был бы невидим, если бы был напрямую включен в исходный код как UTF-8. По возможности избегайте использования префикса u8, так как его поведение значительно изменилось в C++20 по сравнению с C++17. Не стоит использовать типы char16_t и char32_t, так как они предназначены для текстов, не относящихся к UTF-8. По той же причине избегайте wchar_t, если только вы не пишете код для работы с Windows API, где этот тип используется широко.

 

Пробелы и отступы

Выделение фигурных скобок пробелами помогает улучшить читаемость кода. Когда код оформлен с правильными отступами и пробелами, программистам легче понимать структуру и логику программы. Это особенно важно при работе в команде или при чтении чужого кода.

Не рекомендуется:

int x_ = 3, y_ = 7; double z_ = 4.25; x_++;

if (a_ == b_)

{

    Foo();

}

Рекомендуется:

int x_ = 3;

int y_ = 7;

double z_ = 4.25;

x_++;

if (a_ == b_)

{

    Foo();

}

 

Определения функций и объявления

Возвращайте тип функции в той же строке, где указано её имя. Параметры также размещайте на той же строке, если они все умещаются. Если список параметров слишком длинный, переносите его на новые строки так же, как делаете это с аргументами при вызове функции.

Функции оформляются следующим образом:

ReturnType ClassName::FunctionName(Type par_name1_a, Type par_name2_a)

{

    DoSomething();

}

 

Если у вас слишком много текста, чтобы уместиться в одной строке:

ReturnType ClassName::ReallyLongFunctionName(Type par_name1_a,

                                             Type par_name2_a,

                                             Type par_name3_a)

{

    DoSomething();

}

 

 

Или если вы не можете указать даже первый параметр:

ReturnType LongClassName::ReallyReallyReallyLongFunctionName(Type par_name1_a,

                                                             Type par_name2_a,

                                                             Type par_name3_a)

{

    DoSomething();

}

 

Некоторые моменты, на которые следует обратить внимание:

• Выбирайте подходящие имена параметров.

• Имя параметра может быть опущено только в том случае, если параметр не используется в определении функции.

• Если вы не можете уместить возвращаемый тип и имя функции в одной строке, сделайте перерыв между ними.

• Если вы делаете разрыв после возвращаемого типа объявления или определения функции, не делайте отступ.

• Открытая скобка всегда находится в той же строке, что и имя функции. 

• Между именем функции и открытой круглой скобкой никогда не должно быть пробела.

• Между круглыми скобками и параметрами никогда не должно быть пробела.

• Открытая фигурная скобка всегда находится в конце последней строки объявления функции, а не в начале следующей строки.

• Закрывающая фигурная скобка находится либо в последней строке сама по себе, либо в той же строке, что и открывающая фигурная скобка.

• Между закрывающей круглой скобкой и открывающей фигурной скобкой должен быть пробел.

 • Все параметры должны быть по возможности выровнены.

• Отступ по умолчанию составляет 2 пробела.

• Заключенные в оболочку параметры имеют отступ в 4 пробела. Неиспользуемые параметры, которые очевидны из контекста, могут быть опущены:

class Foo

{

public:

    Foo(const Foo&) = delete;

    Foo& operator=(const Foo&) = delete;

};

 

Форматирование пространств имён

Содержимое в пространстве имён пишется без отступа.

Пространство имён не добавляет отступов. Например:

Не рекомендуется:

namespace

{

  // Ошибка! Сделан отступ там, где не нужно

    void Foo()

    {

        …

    }

 

} // namespace

 

Не делайте отступов в пространстве имён:

Рекомендуется:

namespace

{

    void Foo()

    { // Хорошо. Без дополнительного отступа

        ...

    }

} // namespace

 

Форматирование классов

Размещайте секции в следующем порядке: public, protected и private.  

Отступ - четыре пробела. Отделяйте секциями функции от переменных.

Например:
Не рекомендуется:

class MyClass

{

public:

    void Foo();

 

private:

    void Bar();

    int x_ = 0;

    int y_ = 0;

};

Рекомендуется:

class MyClass

{

public:

    void Foo();

 

private:

    void Bar();

 

private:

    int x_ = 0;

    int y_ = 0;

};

Литералы с плавающей точкой

Литералы с плавающей точкой всегда должны иметь исходную точку с цифрами с обеих сторон, даже если они используют экспоненциальную систему счисления. Читаемость улучшается, если все литералы с плавающей точкой принимают эту знакомую форму, поскольку это помогает гарантировать, что их не примут за целочисленные литералы, и что E/e экспоненциальная запись не принимается за шестнадцатеричную цифру. Можно инициализировать переменную с плавающей точкой целочисленным литералом (при условии, что тип переменной может точно представлять это целое число), но обратите внимание, что число в экспоненциальной системе счисления никогда не является целочисленным литералом.

Пример:

Не рекомендуется:

float f_ = 1.f;

long double ld_ = -.5L;

double d_ = 1248e6;  

Рекомендуется:

float f_ = 1.0f;

float f2_ = 1.0;

float f3_ = 1;

long double ld_ = -0.5L;

double d_ = 1248.0e6;

 

Логические выражения

Если у вас есть логическое выражение, длина которого превышает стандартную длину строки, будьте последовательны в разбивке строк. В этом примере логический оператор AND всегда находится в конце строк:

if (this_one_thing_ > this_other_thing_ &&

    a_third_thing_ == a_fourth_thing_   &&

    yet_another_ && last_one_)

{

    ...

}

Обратите внимание, что при переносе кода в этом примере оба && логических оператора AND находятся в конце строки. Хотя перенос всех операторов в начало строки также разрешен. Не стесняйтесь разумно вставлять дополнительные круглые скобки, потому что при правильном использовании они могут быть очень полезны для повышения удобства чтения, но будьте осторожны при чрезмерном использовании. Также обратите внимание, что вы всегда должны использовать операторы пунктуации, такие как && и ~, а не словесные операторы, такие как and и compl.

 

В приведённом примере фигурные скобки отделены пробелами, что делает код визуально разделённым на логические блоки. Такой подход позволяет быстро увидеть, где начинается и заканчивается блок кода, связанный с условием if. Это снижает вероятность ошибок и помогает поддерживать единообразие в стиле написания кода, что важно для больших проектов и коллективной разработки. Ставьте пробелы между операторами и операндами:

int x_ = (a_ + b_) * c_ / d_ + Foo(); Когда строка становится длиннее 100 символов, разделите её на две, сделав перевод на новую строку после оператора, и продолжайте писать:

 int result_ = ReallyLongFunctionOne() + ReallyLongFunctionTwo()

             + ReallyLongFunctionThree() + ReallyLongFunctionFour();

 

int result2_ = ReallyLongFunction(parameter_one_a,

                                  parameter_two_a,

Базовые выражения C++

C++ построен на основе C, поэтому можно решать задачи с помощью C++, так и традиционным способом, используя C. Например, для вывода текста на консоль можно использовать оператор вывода `cout` в стиле C++, или глобальную функцию `printf` в стиле C:

Не рекомендуется:

printf("Hello, world!\n");

Рекомендуется:

std::cout << "Hello, world!" << std::endl;

Если вы сомневаетесь, использовать ли цикл `for` или `while`, помните: используйте `for`, когда количество повторений известно, и `while`, когда оно неизвестно:

Повторяется 'size_' раз:

for (int i_ = 0; i_ < size_; i_++)

{

    ...

}

Повторяется, пока есть ввод:

std::string str_;

while (input_ >> str_)

{

    ...

}

При использовании управляющих операторов, таких как `if / else`, `for`, `while`, всегда оборачивайте тело этих операторов в фигурные скобки `{}` и соблюдайте отступы, даже если тело состоит из одной строки:

Не рекомендуется:

if (size_ == 0)

    return;

else

{

    for (int i_ = 0; i_ < 10; i_++)

        std::cout << "ok" << std::endl;

}

Рекомендуется:

if (size_ == 0)

{

    return;

}

else

{

    for (int i_ = 0; i_ < 10; i_++)

    {

        std::cout << "ok" << std::endl;

    }

}

Избегайте использования `break` и `continue`, за исключением случаев, когда это абсолютно необходимо. Не используйте функцию `exit`, которая завершает программу немедленно. Программа должна завершаться естественным образом через оператор `return` в функции `main`. При использовании операторов `if / else` выбирайте подходящие конструкции в зависимости от условий.

 

Избегайте множественных независимых проверок:

Не рекомендуется:

if (grade_ >= 90)

{

    std::cout << "You got an A!";

}

if (grade_ >= 80 && grade_ < 90)

{

    std::cout << "You got a B!";

}

if (grade_ >= 70 && grade_ < 80)

{

    std::cout << "You got a C!";

}

Рекомендуется:

if (grade_ >= 90)

{

    std::cout << "You got an A!";

}

else if (grade_ >= 80)

{

    std::cout << "You got a B!";

}

else if (grade_ >= 70)

{

    std::cout << "You got a C!";

}

 

Если у вас есть `if / else`, возвращайте результат проверки напрямую:

Не рекомендуется:

if (score1_ == score2_)

{

    return true;

}

else

{

    return false;

}  

Рекомендуется:

return score1_ == score2_;

 

 

 

Не проверяйте значения логического типа, используя `==` или `!=` с `true` или `false`:

Не рекомендуется:

if (x_ == true)

{

else if (x_ != true)

    ...

Рекомендуется:

if (x_)

    ...

Чрезмерность

Если вы используете один и тот же код дважды или более, то найдите способ удалить излишний код, чтобы он не повторялся. К примеру, его можно поместить во вспомогательную функцию. Если повторяемый код похож, но не совсем, то постарайтесь сделать вспомогательную функцию, которая принимает параметры и представляет разнящуюся часть:

Не рекомендуется:

Foo();

x_ = 10;

y_++;

...

Foo();

x_ = 15;

y_++;

Рекомендуется:

Helper(10);

Helper(15);

...

void Helper(int new_x_a)

{

    Foo();

    x_ = new_x_a;

    y_++;

}

Переместите общий код из выражения if / else, чтобы он не повторялся:

Не рекомендуется:

if (x_ < y_)

{

    Foo();

    x_++;

    std::cout << "Привет!";

}

else

{

    Foo();

    y_++;

    std::cout << "Привет!";

}

Рекомендуется:

Foo();

if (x_ < y_)

{

    x_++;

}

else

{

    y_++;

}

std::cout << "Привет!";

Инкапсуляция. Отделяйте ваши объекты, делая все поля данных в вашем классе private:

class Student

private:

.hpp vs .cpp. Всегда размещайте объявления классов и их частей в собственные файлы, ClassName.hpp. Всегда сворачивайте файлы объявления классов .hpp в блок препроцессоров #if defined() / define / endif, чтобы избежать множественных объявлений одного класса:

Point.hpp

#ifndef POINT_HPP

class Point

public:

    int m_y_;

#endif

 

Point.cpp

#include "Point.hpp"

    m_x_ = x_a;

    m_y_ = y_a;

}

 

void Point::Translate(int dx_a, int dy_a)

{

    m_x_ += dx_a;

    m_y_ += dy_a;

}

class vs struct. Всегда используйте class, только если вы не создаете очень маленький и простой вид данных, для которого нужны лишь несколько публичных переменных и, возможно, конструктор для их инициализации. Избегайте лишних полей. Применяйте поля для сохранения значимых сведений о ваших объектах, а не временных значений, применяемых лишь однажды.

 

Функции и процедурное проектирование

Хорошо разработанная функция должна обладать следующими качествами:

v Выполняет конкретную задачу полностью и без отклонений.

v Не берёт на себя лишней работы.

v Не взаимодействует с другими функциями без необходимости.

v Эффективно использует память для хранения данных.

v Помогает структурировать программу и сделать её более ясной.

v Способствует избавлению от ненужного кода, который иначе мог бы появиться.

Используйте параметры для передачи данных в функцию или возврата нескольких значений. Не добавляйте параметры, если в них нет необходимости. Например, в следующей функции переменные `a`, `b` и `c` не являются параметрами, потому что это не требуется:

void Quadratic(double a_a, double b_a, double c_a, double& root1_a,

               double& root2_a)

{

    double d_ = sqrt(b_a * b_a - 4 * a_a * c_a);

    root1_a = (-b_a + d_) / (2 * a_a);

    root2_a = (-b_a - d_) / (2 * a_a);

}

Если нужно вернуть значение, предпочтительно использовать `return`:

Не рекомендуется:

void Max(int a_a, int b_a, int& result_a)

{

    result_a = (a_a > b_a) ? a_a : b_a;

}

Рекомендуется:

int Max(int a_a, int b_a)

{

    return (a_a > b_a) ? a_a : b_a;

}

 

Передавая объекты в функцию, старайтесь использовать ссылки, так как это экономит память, избегая копирования объектов. Отдавайте предпочтение ссылочным переменным вместо указателей, поскольку ссылки не могут быть `NULL`:

Не рекомендуется:

void Process(BankAccount* account_a)

{

    ...

}

Рекомендуется:

void Process(BankAccount& account_a)

{

    ...

}

Если объект не будет изменяться в функции, передавайте его как `const`- ссылку:

Не рекомендуется:

void Display(BankAccount account_a)

{

    ...

}

Рекомендуется:

void Display(const BankAccount& account_a)

{

    ...

}

 

Избегайте длинных цепочек вызовов функций, когда одна функция вызывает другую, а та — следующую, не возвращая значения в `main`. Стремитесь к тому, чтобы `main` давала общее представление о программе.
