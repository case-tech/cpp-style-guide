Introduction

C++ is a compiled, object-oriented, general-purpose programming language. It is used by a wide range of organizations around the world. When developing programs in C++, it is important to follow a consistent coding style to ensure that the code is easy to debug, read, and maintain. The purpose of this guide is to provide an overview of the coding practices we follow when developing software in C++ and why we believe they are correct. It is important to note that this is not a comprehensive C++ tutorial, and we assume that the reader is already familiar with the language.

 

Version C++

As of 2025, the code should be written using the features of C++ 20. It is consistently supported by a larger number of compilers than C++ 23. For comparison, refer to the standard tables on the website cppreference.

Try to minimize the use of non-standard language extensions in your code. These include GCC attribute, built-in functions such as __builtin_prefetch or SIMD, #pragma directives, inline assembly, macros such as COUNTER or PRETTY_FUNCTION, compound expressions (for example, foo = ({ int x; Bar(&x); x })), variable-length arrays, functions such as alloca(), and the a?.

 

Although non-standard extensions may provide useful features that are not present in standard C++ and may be necessary to optimize compiler performance, they are not always supported by all compilers. Using such extensions reduces the portability of the code, as they may work differently in different compilers. Even if an extension is supported by all target compilers, its implementation may differ, leading to ambiguities in behavior. The use of non-standard extensions complicates the code, requiring the developer to know these extensions in order to understand it and transfer it to other architectures. Therefore, do not use non-standard extensions, except when you use them in wrappers that provide portability. In this case, such wrappers should be included in a dedicated header for the entire project.

 

Comments

Comments are extremely important for maintaining code readability. Here are some basic rules for writing them. However, remember that while comments are very useful, the best code is the code that explains itself. It is better to use clear names for types and variables than to explain unclear names in comments. When writing comments, focus on your audience: the people who will be reading your code later. Be generous with your comments РІР‚вЂќ you might be reading them yourself in the future!

 

Punctuation, spelling, and grammar

Pay attention to the punctuation, spelling, and grammar in your comments. Well-written comments are easier to read than poorly written ones. Comments should be easy to read, like regular text, with proper use of capital letters and punctuation. Full sentences are usually easier to read than fragments of phrases. Short comments, such as notes at the end of a code line, can be less formal, but try to maintain a consistent style. While it may not be pleasant when a reviewer points out a punctuation error, it's important to keep the code clear and understandable. Proper use of punctuation, spelling, and grammar contributes to this.

 

Comment style

You can use both single-line comments // and multi-line comments /* */, but it is important to maintain consistency. Although both styles of comments are acceptable, // is more commonly used. Be consistent in choosing the comment style and applying it to your code.

 

Comments on files

When creating each header file, use the license template. If the source file (for example, an .hpp file) contains several custom abstractions (such as common functions or related classes), add a comment that describes these abstractions. The comment should be detailed enough so that future developers can understand what is contained in the file. However, more detailed documentation on individual abstractions should be found directly in their descriptions, and not in the comments to the file. For example, if you are writing a comment to the Frobber.hpp file, you do not need to add a similar comment to the files. Frobber.cpp or Frobber.test.hpp. At the same time, if you have a file, for example RegisteredObjects.cpp containing a collection of classes without an associated header file, add a comment to this file to explain its contents.

 

The official notification and the author's line

Each header file must contain a license template. Make sure you use the correct template for the license used in your project (e.g., Apache 2.0, BSD, LGPL, GPL). If you make significant changes to a file that includes "Author", consider removing this line. New files typically do not require copyright notices or "Author" lines.

 

Comments on issues

Use TODO comments for temporary solutions, short-term fixes, or for cases where the code is not perfect yet, but it works. TODO comments should contain the word TODO in capital letters, as well as the error ID, name, email address, or other communication method that will help you better understand the problem.

Examples:

// TODO: Error 12345678 - delete after the window expires

// compatibility 2047q4.

// TODO (Ivan): Use "\*" here to concatenate strings.

 

If your TODO includes a plan for the future, specify either a specific date ("fix by November 2025") or an event ("remove this code when all clients can handle XML responses").

 

Naming

The most important consistency rules are those that govern naming. The name style immediately tells us what kind of named object it is: type, variable, function, constant, macro, etc., without requiring us to search for the declaration of this object. The mechanism of selecting patterns in our brain largely depends on these naming rules. The naming rules are pretty arbitrary, but we believe that consistency in this area is more important than individual preferences, so whether you consider them reasonable or not, rules are rules.

 

File Names

All file names must be written in Camel Case. Follow the rules used in your project. If there is no consistent local pattern to follow, give preference to this case. Example of a valid file name: MyUsefulClass.cpp

 

C++ files must end in .cpp, and header files must end in .hpp. Files that rely on text inclusion in certain locations must end in .inc. In general, make the file names very specific. For example, use HttpServerLogs.hpp instead of Logs.hpp. A very common case is the presence of a pair of files named, for example, FooBar.hpp and FooBar.cpp , defining a class named FooBar.

 

Mixing C and C++ header files can lead to confusion:

Formatting the code.

The IDE often adjusts formatting for .h and .hpp files are different. If used .h for C++ headers, you can accidentally apply C styling.

Confusion in naming.

C and C++ headers with similar names (for example, MyClass.h and MyClass.hpp) help you quickly distinguish between C and C++ files, which is especially important when using C wrappers for libraries.

Errors when connecting C headers in C++.

When linking C headers, you need to wrap them in extern "C" to avoid call conflicts due to different name mangling. If you don't do this, you may encounter compilation errors.

 

Type Names

Type names begin with a capital letter and contain a capital letter for each new word without an underscore: MyExcitingClass, MyExcitingEnum. All type namesРІР‚вЂќclasses, structures, type aliases, enumerations, and type template parametersРІР‚вЂќfollow the same naming convention. Type names must begin with a capital letter and contain a capital letter for each new word. There are no underscores.

Examples:

// Classes and Structs

class UrlTable{};

class UrlTableTester{};

struct UrlTableProperties{};

 

// Typedefs

typedef hash_map<UrlTableProperties*, std::string> PropertiesMap;

 

// using aliases

using PropertiesMap = hash_map<UrlTableProperties*, std::string>;

 

// enums

enum class UrlTableError{};

 

Function names

The names of public functions of a class or structure begin with a capital letter, each word in the name also begins with a capital letter, underscores are not used: MyFunction(). The names of private functions of a class or structure begin with the character "_", the rest of the name is formed similarly to public functions. If the function is outside the class, it should be referred to as public.

 

Names of function arguments and variables

The words in the function argument names must always be written in lowercase, separated by the "_", and end with "_a". Similarly, variables end with "_".

 

Formatting

Coding style and formatting do not have strict standards, but using a consistent style makes it easier to maintain control over the project. Team members may have different preferences for formatting rules, and some may require adaptation, but it is important that everyone follows a consistent style so that everyone can easily read and understand each other's code.

 

Line length

Lines of code should not exceed 80 characters, as this is the standard that most codes follow. Proponents of the rule claim that it helps when working with multiple windows where there is no room for expansion. Opponents believe that the 80-character limit is outdated and makes the code less readable on modern screens. Exceptions are allowed for long comments, string literals, include instructions, header protection, and using declarations that cannot be broken down without loss of convenience. In such cases, Clang-Format may be disabled.

 

Characters other than ASCII

Characters that are not included in the ASCII set should be used rarely and only in UTF-8 format. It is advisable not to embed the user's text directly into the source code, even if it is in English, so characters other than ASCII should be used minimally. However, in some cases it may be justified. For example, if your code works with data from files from foreign sources, it makes sense to encode non-ASCII strings if they are used as delimiters in this data. Similarly, test codes that do not need localization may include such strings. In such situations, it is better to use UTF-8, since this encoding is supported by most tools that work not only with ASCII. It is acceptable and even recommended to use hexadecimal encoding where it improves readability. For example, "\XeF\xBB\xBF" or "\uFEFF" denote a zero-width character without a Unicode break, which would be invisible if it were directly included in the source code as UTF-8. If possible, avoid using the u8 prefix, as its behavior has changed significantly in C++20 compared to C++17. You should not use the char16_t and char32_t types, as they are intended for non-UTF-8 texts. For the same reason, avoid wchar_t unless you are writing code to work with the Windows API, where this type is widely used.

 

Spaces and indents

Separating curly braces with spaces helps to improve the readability of the code. When the code is designed with the correct indents and spaces, it is easier for programmers to understand the structure and logic of the program. This is especially important when working in a team or when reading someone else's code.

Not recommended:

if (a_ == b_)

{

    Foo();

}

Recommended:

int y_ = 7;

x_++;

{

    Foo();

}

 

Function definitions and declarations

Returns the type of the function in the same line where its name is specified. Also place the parameters on the same line if they all fit. If the list of parameters is too long, move it to new lines in the same way as you do with arguments when calling a function.

The functions are designed as follows:

Floating-point literals

Floating-point literals should always have a starting point with digits on both sides, even if they use an exponential number system. Readability is improved if all floating-point literals take this familiar form, as it helps ensure that they are not mistaken for integer literals, and that the E/e exponential notation is not mistaken for a hexadecimal digit. You can initialize a floating-point variable with an integer literal (provided that the type of the variable can accurately represent this integer), but note that a number in the exponential notation is never an integer literal.

Example:

Not recommended:

float f_ = 1.f;

long double ld_ = -.5L;

double d_ = 1248e6;  

Recommended:

float f_ = 1.0f;

float f2_ = 1.0;

float f3_ = 1;

long double ld_ = -0.5L;

double d_ = 1248.0e6;

 

Logical expressions

If you have a boolean expression that is longer than the standard string length, be consistent in breaking down the strings. In this example, the logical AND operator is always at the end of the lines.:

Also note that you should always use punctuation operators such as && and ~, rather than verbal operators such as and and compl.

Basic C++ Expressions

C++ is built on the basis of C, so you can solve problems using C++, and in the traditional way, using C. For example, to output text to the console, you can use the C++-style `cout` output operator, or the C-style global `printf` function:

Not recommended:

Recommended:

std::cout << "Hello, world!" << std::endl;

 

If you are in doubt whether to use the `for` or `while` loop, remember: use `for` when the number of repetitions is known, and `while` when it is unknown:

Repeats 'size_' times:

{

    ...

}

Repeats as long as there is input:

while (input_ >> str_)

{

    ...

}

When using control operators such as `if/else`, `for`, `while`, always wrap the body of these operators in curly braces `{}` and observe indentation, even if the body consists of a single line:

Not recommended:

    return;

else

{

    for (int i_ = 0; i_ < 10; i_++)

        std::cout << "ok" << std::endl;

}

Recommended:

{

    return;

}

else

{

    for (int i_ = 0; i_ < 10; i_++)

    {

        std::cout << "ok" << std::endl;

    }

}

Avoid using `break` and `continue`, except when absolutely necessary. Do not use the `exit` function, which terminates the program immediately. The program should terminate naturally via the 'return` operator in the `main' function. When using the 'if/else` operators, choose the appropriate constructions depending on the conditions.

Functions and procedural design

A well-designed function should have the following qualities:

v Performs a specific task completely and without deviations.

v Does not take on unnecessary work.

v Does not interact with other functions unnecessarily.

v Efficiently uses memory to store data.

v Helps to structure the program and make it more clear.

v Helps to eliminate unnecessary code that might otherwise appear.

Use parameters to pass data to a function or return multiple values. Do not add parameters unless necessary. For example, in the following function, the variables `a`, `b`, and `c` are not parameters because they are not required:

void Quadratic(double a_a, double b_a, double c_a, double& root1_a,

               double& root2_a)

{

    double d_ = sqrt(b_a * b_a - 4 * a_a * c_a);

    root1_a = (-b_a + d_) / (2 * a_a);

    root2_a = (-b_a - d_) / (2 * a_a);

}

If you need to return a value, it is preferable to use `return`:

Not recommended:

void Max(int a_a, int b_a, int& result_a)

{

    result_a = (a_a > b_a) ? a_a : b_a;

}

Recommended:

int Max(int a_a, int b_a)

{

    return (a_a > b_a) ? a_a : b_a;

}

 

When passing objects to a function, try to use references, as this saves memory by avoiding copying objects. Give preference to reference variables instead of pointers, since references cannot be NULL:

Not recommended:

void Process(BankAccount* account_a)

{

    ...

}

Recommended:

void Process(BankAccount& account_a)

{

    ...

}

 

If the object will not be modified in the function, pass it as a `const` reference:

Not recommended:

void Display(BankAccount account_a)

{

    ...

}

Recommended:

void Display(const BankAccount& account_a)

{

    ...

}

 

Avoid long chains of function calls, when one function calls another, and that calls the next one, without returning values to `main'. Aim for the `main` to give a general idea of the program.
